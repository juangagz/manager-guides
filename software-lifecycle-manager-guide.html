Software Development Lifecycle

Lo que todo manager no-dev necesita saber para hablar con su equipo

El ciclo de vida del software (SDLC)

software development life cycle — el mapa general

Todo software pasa por las mismas fases, sin importar el tamaño de la empresa. Lo que cambia es qué tan formal, rápido, o iterativo es el proceso.

01
Planeación y Requerimientos

¿Qué problema resuelve? ¿Para quién? Se define el alcance, los stakeholders, y las restricciones. El output es un documento de requerimientos o un backlog inicial.

02
Diseño

Arquitectura del sistema, diseño de base de datos, flujos de UI/UX. Los devs senior y arquitectos deciden cómo se va a construir antes de escribir código.

03
Desarrollo (Coding)

Los developers escriben el código según el diseño. Es la fase más visible pero no siempre la más larga.

04
Testing / QA

Se verifica que el software funcione como se esperaba. Incluye pruebas funcionales, de regresión, de carga y más. (Ver sección dedicada.)

05
Deployment

El software se instala en producción. Puede ser un "release" grande o deployments continuos varias veces al día.

06
Mantenimiento y Operaciones

Una vez en producción, el software necesita monitoreo, corrección de bugs, y mejoras. Esta fase nunca termina mientras el producto esté activo.

Para el manager Tu trabajo en este ciclo es asegurar que las transiciones entre fases sean claras: ¿quién aprueba que algo está listo para pasar a la siguiente fase? ¿Hay criterios de salida ("definition of done")?

Metodologías de desarrollo

cómo se organiza el trabajo en la práctica
WATERFALL (Cascada)
  • Fases secuenciales: una termina antes de que empiece la siguiente
  • Mucha documentación al inicio
  • Ideal para proyectos con requerimientos muy fijos (ej. contratos gubernamentales)
  • Si algo cambia a mitad de camino, es costoso
  • Entrega al final del proyecto
AGILE (Ágil)
  • Iterativo: ciclos cortos (sprints) de 1–4 semanas
  • Se entrega valor incremental en cada ciclo
  • Requerimientos pueden cambiar entre sprints
  • Más colaboración continua con el cliente/usuario
  • La mayoría de empresas de software lo usan hoy

Scrum — el framework ágil más común

Scrum organiza el trabajo en Sprints. Los eventos clave que tu equipo probablemente hace:

EventoQué esTu rol como manager
Sprint PlanningSeleccionar qué se hará en el sprint (1–2 semanas)Asegurar que las prioridades de negocio estén reflejadas
Daily StandupReunión diaria de 15 min: ayer / hoy / bloqueosEscuchar bloqueos, no micro-gestionar
Sprint ReviewDemo de lo que se construyó al stakeholderDar feedback real, validar si cumple expectativas
RetrospectivaEl equipo reflexiona: qué salió bien / qué mejorarCrear espacio seguro para feedback honesto

Kanban — flujo continuo

Alternativa a Scrum. No hay sprints fijos; las tareas fluyen de "To Do → In Progress → Done". Bueno para equipos de soporte o maintenance. El tablero (Jira, Trello, Linear) es la herramienta central.

Vocabulario clave Backlog: lista priorizada de todo lo que el equipo podría construir.
User Story: requerimiento escrito desde la perspectiva del usuario ("Como usuario quiero… para poder…").
Velocity: cuánto trabajo completa un equipo por sprint. Sirve para estimar fechas.

QA vs Testing vs Automation

la confusión más común en equipos de software
Concepto amplio

QA — Quality Assurance

Es el proceso completo de garantizar calidad. No solo probar: incluye revisar requerimientos, procesos, estándares, y prevenir defectos antes de que ocurran. Es una cultura, no solo un rol.

Actividad específica

Testing (QC)

Es la ejecución de pruebas sobre el software construido. Detecta bugs. Es parte de QA, pero QA es más amplio. "Testing es encontrar bugs; QA es no tener que encontrarlos."

Herramienta / técnica

Automation Testing

Testing ejecutado por scripts, no por humanos. Más rápido, repetible, ideal para regressions. Requiere inversión inicial para crear los scripts, pero ahorra tiempo a largo plazo.

Tipos de pruebas — de más básico a más complejo

TipoQué pruebaQuién lo hace
Unit Test Una función o componente aislado. "¿Esta función hace lo que dice?" El mismo developer
Integration Test Cómo se comunican dos o más partes del sistema juntas Developers / QA
End-to-End (E2E) Flujo completo del usuario, de inicio a fin. "El usuario puede comprar un producto" QA / Automation
Regression Test Verificar que cambios nuevos no rompieron algo que ya funcionaba Idealmente automatizado
Performance / Load Test ¿Qué pasa si 10,000 usuarios entran al mismo tiempo? QA especializado / DevOps
UAT (User Acceptance Test) El cliente o usuario final valida que cumple sus necesidades reales Negocio / cliente
Smoke Test Prueba rápida post-deploy: "¿Lo más básico funciona?" QA / automatizado

La Pirámide de Testing

Un principio guía: tener muchos unit tests (baratos, rápidos), menos integration tests, y pocos E2E (costosos, lentos). Si tu equipo tiene solo tests E2E, hay un problema de arquitectura.

Señal de alerta como manager Si el equipo dice "no tenemos tiempo para tests", eso es deuda técnica acumulándose. Los bugs en producción son mucho más caros que invertir en testing antes. Pregunta: ¿qué porcentaje del código tiene cobertura de tests (test coverage)?

Roles en un equipo de software

quién hace qué — y a quién preguntarle qué
Desarrollo

Frontend Developer

Construye lo que el usuario ve: interfaces, botones, flujos visuales. Trabaja con HTML, CSS, JavaScript / frameworks como React o Vue.

Desarrollo

Backend Developer

Construye la lógica del servidor, APIs, conexión a base de datos. El usuario no lo ve directamente, pero es lo que hace funcionar todo.

Desarrollo

Full Stack

Hace ambos. Común en startups o equipos pequeños. En empresas grandes suele especializarse con el tiempo.

Infraestructura

DevOps / SRE

Gestiona pipelines de CI/CD, infraestructura en cloud, monitoreo, y disponibilidad. "El que hace que el código llegue a producción de forma confiable."

Calidad

QA Engineer

Diseña y ejecuta estrategias de testing. En equipos modernos escribe tests automatizados. Puede especializarse en performance, seguridad, etc.

Diseño

UX/UI Designer

Diseña cómo funciona (UX) y cómo se ve (UI) el producto. El output son wireframes, prototipos en Figma. Trabaja antes de que el dev empiece a codear.

Producto

Product Manager (PM)

Define qué construir y por qué. Es el dueño del backlog y el puente entre negocio y equipo técnico. No es lo mismo que un Project Manager.

Liderazgo técnico

Tech Lead / Architect

Developer senior que guía decisiones técnicas: qué tecnologías usar, cómo está estructurado el sistema, revisión de código (code review).

Confusión clásica: PM vs Project Manager Product Manager define QUÉ construir (prioridades, visión del producto). Project Manager gestiona CÓMO se ejecuta (tiempos, recursos, riesgos del proyecto). En muchas empresas el mismo rol hace ambos.

Entornos de despliegue

dev / staging / production — por qué existen

El código no va directo de la cabeza del developer al usuario. Pasa por entornos que aíslan riesgos.

DEV
Development (Local / Dev)

El developer trabaja en su máquina o un servidor de desarrollo. Nada aquí afecta a usuarios reales. Puede estar roto en cualquier momento.

STG
Staging / Pre-producción / QA

Espejo de producción con datos de prueba. Aquí QA hace sus pruebas. Los stakeholders pueden revisar antes del release. Lo que funciona aquí, debería funcionar en producción.

PROD
Production

El entorno real. Usuarios reales, datos reales, dinero real. Un bug aquí tiene impacto directo en el negocio. Los deployments a producción deben ser controlados.

CI/CD — la tubería que mueve el código

CI — Continuous Integration
  • Cada vez que un dev sube código, se ejecutan tests automáticamente
  • Detecta conflictos entre el trabajo de diferentes devs
  • Si los tests fallan, el merge se bloquea
  • Herramientas: GitHub Actions, GitLab CI, Jenkins
CD — Continuous Delivery/Deployment
  • Automatiza el deploy a staging o producción
  • Delivery: el deploy requiere aprobación manual
  • Deployment: el deploy es 100% automático si pasan los tests
  • Empresas maduras hacen múltiples deploys diarios
Lo que debes saber como manager Si tu equipo no tiene CI/CD, cada deploy es manual y riesgoso. Pregunta: ¿cuánto tiempo toma ir de código aprobado a producción? Si la respuesta es "días", hay una oportunidad de mejora.

Deuda técnica y otros conceptos de salud del sistema

lo que tu equipo dice y lo que realmente significa
TérminoQué significa para tu equipoQué significa para el negocio
Deuda técnica Código escrito rápido o sin las mejores prácticas, que habrá que refactorizar después Cada sprint futuro cuesta más. Nuevas features tardan más. Más bugs.
Refactoring Reescribir código existente para hacerlo más limpio, sin cambiar lo que hace Inversión en velocidad futura. No produce features visibles a corto plazo.
Bug / Defecto El software hace algo que no debería, o no hace algo que debería Puede ser crítico (sistema caído) o menor (error visual). Prioridad depende del impacto.
Hotfix Fix urgente en producción que se salta el proceso normal Necesario a veces, pero si ocurre seguido, hay problemas de calidad.
Scope creep Requerimientos que crecen en medio del desarrollo sin ajustar plazo ni recursos La causa #1 de proyectos retrasados. Necesitas control de cambios.
Code review Un developer revisa el código de otro antes de mergearlo Reduce bugs, distribuye conocimiento, mantiene estándares. No es burocracia.
Feature flag Un switch que activa/desactiva una feature sin hacer deploy nuevo Permite lanzar a un subgrupo de usuarios primero, o apagar algo rápido.
Monitoring / Observability Herramientas que muestran qué está pasando en producción en tiempo real Sin esto, te enteras de los problemas por los usuarios, no por el sistema.
La pregunta que debes hacer una vez al trimestre "¿Qué porcentaje de nuestro tiempo lo gastamos en deuda técnica vs features nuevas?" Un equipo sano dedica ~20% a mejoras técnicas. Si es 0%, la velocidad bajará con el tiempo.

Quiz rápido

valida lo que absorbiste

Tu equipo dice "necesitamos hacer regression testing antes del release". ¿Qué significa?

Los regression tests protegen lo que ya funciona. Son especialmente importantes en sistemas grandes donde un cambio puede tener efectos inesperados en partes aparentemente no relacionadas.

¿Cuál es la diferencia entre QA y Testing?

QA incluye revisar requerimientos, procesos, y prevenir defectos. Testing es encontrarlos una vez que el código existe. Un buen equipo de QA hace que el testing sea menos necesario.

El equipo pide tiempo para "refactoring". Como manager, ¿cómo lo interpretas?

Negar el refactoring genera más deuda técnica. El código que no se mantiene se vuelve más costoso de cambiar con el tiempo, frenando las nuevas features.

¿Qué hace un DevOps engineer que un Backend developer no hace?

DevOps se ocupa del "cómo llega el código a producción de forma confiable". El Backend developer se ocupa de "qué hace el código cuando ya está ahí".

Tu equipo tiene un "hotfix en producción" todos los viernes. ¿Qué señal da esto?

Los hotfixes ocasionales son normales. Frecuentes sugieren que el proceso de QA no está filtrando problemas antes del release. Es una señal de deuda técnica o falta de automation.
