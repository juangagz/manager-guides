El ciclo de vida de un proyecto de AI/ML

machine learning lifecycle — cómo se construye de verdad

A diferencia del software tradicional, en AI el ciclo no es lineal: los equipos vuelven atrás constantemente — si el modelo no funciona, puede ser problema de los datos, del diseño, o del planteamiento del problema.

Haz clic en cada fase para expandir.

01
Definición del problema Negocio
¿Qué decisión o tarea se quiere automatizar o mejorar? Aquí se define si el problema es de clasificación (¿spam o no?), regresión (¿cuánto vale?), generación (¿qué texto produce?), u otro tipo. Un problema mal definido produce un modelo que nadie usa, aunque funcione perfectamente.
📌 Pregunta que debes hacer"¿Cómo medimos el éxito de este modelo en términos de negocio — no solo de métricas técnicas?"
02
Recolección y exploración de datos Datos
Se recopilan los datos necesarios (bases de datos, APIs, web scraping, sensores, etc.) y se hace un EDA (Exploratory Data Analysis): entender la distribución, buscar valores faltantes, detectar outliers, entender qué hay disponible. Los datos raramente están listos para usar.
📌 Pregunta que debes hacer"¿Tenemos suficientes datos? ¿Son representativos del problema real? ¿Hay sesgos que podamos heredar?"
03
Preparación y Feature Engineering Datos
La fase más larga (60–80% del tiempo en proyectos reales). Incluye limpieza de datos, normalización, transformación, y feature engineering: crear las variables (features) que el modelo usará para aprender. Una variable bien construida puede ser más valiosa que un modelo más sofisticado.
📌 Señal de alertaSi el equipo quiere ir directo al modelo sin pasar tiempo en datos, hay riesgo. "Garbage in, garbage out" — si los datos son malos, el modelo será malo.
04
Entrenamiento del modelo Modelo
Se selecciona un algoritmo (regresión, árbol de decisión, red neuronal, etc.) y se entrena con los datos preparados. El modelo aprende patrones ajustando parámetros internos. Se prueban múltiples experimentos con distintos algoritmos e hiperparámetros (configuraciones del modelo) para encontrar la mejor combinación.
📌 Vocabulario claveTraining set: datos para entrenar. Validation set: datos para ajustar durante el proceso. Test set: datos que el modelo nunca vio, para evaluación final. Este split es crítico para que los resultados sean honestos.
05
Evaluación del modelo Modelo
Se mide qué tan bien funciona el modelo con métricas específicas del problema. Accuracy (exactitud) no siempre es la métrica correcta — en fraude bancario, detectar el 1% de casos falsos positivos puede ser más importante que el accuracy general. La evaluación también incluye buscar bias (sesgos) en las predicciones.
📌 Pregunta que debes hacer"¿Qué métrica estamos optimizando y por qué? ¿Coincide con el objetivo de negocio que definimos al inicio?"
06
Deployment del modelo MLOps
El modelo se integra en un sistema de producción — puede ser una API que otros sistemas llaman, un proceso batch que corre noche, o un componente embebido en una app. A diferencia del software tradicional, un modelo en producción puede degradarse solo con el tiempo (model drift) sin que nadie cambie el código.
📌 Señal de alertaUn modelo que vive solo en un notebook de Jupyter no está "en producción". Pregunta: ¿cómo llega la predicción al usuario o al sistema que la necesita?
07
Monitoreo y Reentrenamiento MLOps
El trabajo no termina con el deploy. Los modelos se monitorizan para detectar data drift (los datos de entrada cambian) o concept drift (la relación entre variables y resultado cambia). Cuando el desempeño baja, el modelo debe ser reentrenado con datos más recientes.
📌 Pregunta que debes hacer"¿Con qué frecuencia reentrenamos? ¿Quién es el dueño de monitorear que el modelo sigue funcionando bien en producción?"

Datos y Pipelines

la base de todo proyecto de ai — lo que nadie ve pero todo depende

¿Qué es un data pipeline?

Una secuencia automatizada de pasos que mueve, transforma y carga datos de una fuente a un destino. En AI, el pipeline alimenta al modelo. Si el pipeline falla, el modelo falla.

ETL (Extract · Transform · Load)
  • Se transforma antes de cargar al destino
  • Tradicional, usado en Data Warehouses
  • Los datos llegan limpios y estructurados
  • Herramientas: SSIS, Informatica, ADF
ELT (Extract · Load · Transform)
  • Se carga crudo y se transforma después
  • Moderno, usado en Data Lakes / Lakehouses
  • Más flexible para AI y exploración
  • Herramientas: dbt, Spark, Databricks

Tipos de datos en AI

TipoEjemploUso en AI
EstructuradoTablas SQL, CSV, ExcelModelos de predicción clásicos (XGBoost, regresión)
No estructuradoTexto, imágenes, audio, videoNLP, visión por computadora, LLMs
Semi-estructuradoJSON, XML, logsPipelines de eventos, análisis de APIs
Time seriesMétricas de sensores, preciosForecasting, detección de anomalías

Calidad de datos — el tema que más retrasa proyectos

Problema

Datos faltantes

Valores nulos o incompletos. El equipo debe decidir si imputar (rellenar), eliminar, o diseñar el modelo para tolerarlos.

Problema

Datos desbalanceados

Cuando una clase es muy rara (ej: fraude = 0.1% de transacciones). El modelo puede predecir "no fraude" siempre y tener 99.9% de accuracy sin detectar nada.

Problema

Data leakage

Información del futuro que "filtra" al training. El modelo parece perfecto en pruebas pero falla en producción. Error frecuente y costoso.

Práctica

Data versioning

Igual que el código tiene versiones (Git), los datasets también deben versionarse. Si no sabes con qué datos se entrenó un modelo, no puedes reproducirlo ni auditarlo.

Para el manager "¿Tenemos un catálogo de datos (data catalog)?" Si no saben qué datos tienen, dónde están, y quién es el dueño de cada fuente, el proyecto de AI tardará más de lo esperado.

Modelos y Evaluación

qué es un modelo, cómo se mide, y cuándo confiar en él

¿Qué es un modelo de ML?

Un modelo es una función matemática que aprendió patrones de datos para hacer predicciones o tomar decisiones. No es magia: es estadística y álgebra aplicada. Cuando el equipo dice "el modelo predice", significa "esta función, dados estos inputs, devuelve este output".

Tipos de aprendizaje

Supervisado
  • Se entrena con datos etiquetados (inputs + respuesta correcta)
  • Aprende a predecir la etiqueta para datos nuevos
  • Ejemplos: clasificar emails, predecir precios, detectar fraude
  • Requiere mucho tiempo de etiquetado humano
No supervisado
  • Se entrena con datos sin etiquetas
  • Encuentra patrones y agrupaciones por sí solo
  • Ejemplos: segmentación de clientes, detección de anomalías
  • Más difícil de evaluar — no hay "respuesta correcta"
Reinforcement Learning
  • Un agente aprende por ensayo y error con recompensas
  • No necesita datos etiquetados, aprende interactuando
  • Base de sistemas como AlphaGo y ChatGPT (RLHF)
  • Complejo de implementar, pero poderoso
Transfer Learning
  • Tomar un modelo ya entrenado y adaptarlo a un problema nuevo
  • Ahorra enormes cantidades de datos y tiempo
  • Base del fine-tuning de LLMs
  • Lo más común hoy en proyectos con modelos de lenguaje

Métricas de evaluación — el vocabulario que necesitas

MétricaQué mideCuándo importa más
Accuracy% de predicciones correctas en totalCuando las clases están balanceadas
PrecisionDe lo que predijo positivo, ¿cuánto era realmente positivo?Cuando los falsos positivos son costosos (ej: spam que bloquea emails reales)
RecallDe todos los positivos reales, ¿cuántos detectó?Cuando los falsos negativos son costosos (ej: no detectar una enfermedad)
F1 ScorePromedio ponderado de Precision y RecallCuando hay desbalance y ambos errores importan
AUC-ROCQué tan bien separa el modelo las clasesClasificación binaria con umbral ajustable
MAE / RMSEError promedio en predicciones numéricasForecasting, predicción de precios, regresión
LatenciaCuánto tarda el modelo en responderSiempre en sistemas en tiempo real

Overfitting vs Underfitting

Overfitting — sobreajuste
  • El modelo memorizó los datos de entrenamiento
  • Funciona muy bien en pruebas, muy mal en producción
  • Como estudiar solo las respuestas del examen anterior
  • Solución: más datos, regularización, simplificar el modelo
Underfitting — subajuste
  • El modelo es demasiado simple para capturar los patrones
  • Funciona mal en training y en producción
  • Como intentar predecir el precio de casas solo con el color
  • Solución: modelo más complejo, más features, más datos
Alerta de manager Si el equipo reporta métricas solo del training set (datos con los que entrenó), desconfía. La única evaluación que importa es sobre datos que el modelo nunca vio.

MLOps y Producción

machine learning operations — cómo los modelos llegan y sobreviven en producción

MLOps es la disciplina que une ML con DevOps. Su objetivo: llevar modelos a producción de forma confiable, reproducible y monitoreable. Sin MLOps, los proyectos de AI quedan atrapados en notebooks que nadie más puede usar.

Sin MLOps (caos)
  • "El modelo funciona en mi máquina"
  • Nadie sabe con qué datos se entrenó el modelo en producción
  • Reproducir un experimento tarda días
  • Cuando el modelo falla, nadie sabe por qué
  • El data scientist es el único que puede actualizar el modelo
Con MLOps (control)
  • Pipeline automatizado: datos → entrenamiento → deploy
  • Experimentos registrados (métricas, parámetros, datos)
  • Cualquier modelo puede ser reproducido o rollback
  • Monitoreo automático de desempeño
  • Reentrenamiento automatizado al detectar drift

Conceptos clave de MLOps

Registro

Model Registry

Un repositorio centralizado donde se guardan los modelos entrenados con sus métricas, versiones, y metadatos. Como un GitHub pero para modelos. Herramientas: MLflow, W&B, Azure ML.

Seguimiento

Experiment Tracking

Registrar automáticamente cada experimento: qué hiperparámetros se usaron, qué datos, qué métricas resultaron. Sin esto, el equipo repite experimentos sin saberlo.

Despliegue

Model Serving

Exponer el modelo como una API para que otros sistemas lo consuman. El modelo recibe inputs en tiempo real y devuelve una predicción. Herramientas: FastAPI, TorchServe, Triton.

Vigilancia

Model Monitoring

Detectar cuándo el modelo empieza a funcionar peor. Puede ser por data drift (los inputs cambian) o concept drift (la relación entre variables cambia en el mundo real).

Empaquetado

Feature Store

Repositorio centralizado de features calculadas y reutilizables. Evita que cada equipo recalcule las mismas variables. Reduce inconsistencias entre entrenamiento y producción.

Automatización

CI/CD for ML

Pipelines que automatizan reentrenamiento, validación y re-deploy al detectar drift o nuevos datos. El equivalente del CI/CD de software, pero para modelos.

Madurez de MLOps Nivel 0: Todo manual, notebooks en laptops personales.
Nivel 1: Pipelines de entrenamiento automatizados.
Nivel 2: CI/CD completo — desde datos hasta producción, sin intervención humana.

La mayoría de las empresas está en nivel 0–1. Llegar a nivel 2 es un proyecto en sí mismo.

GenAI, LLMs y Agentes

el mundo de la ia generativa — lo que cambió todo desde 2022

Los Large Language Models (LLMs) son modelos entrenados con enormes cantidades de texto para generar, resumir, traducir, razonar, y responder preguntas. A diferencia del ML clásico, no predices un número — generas contenido. GPT-4, Claude, Gemini, Llama son ejemplos.

Formas de usar un LLM en un producto

TécnicaQué esCuándo usarlaCosto relativo
Prompting Usar el LLM directamente con instrucciones bien escritas Tareas generales, prototipado rápido Bajo
Few-shot Dar ejemplos dentro del prompt para guiar el formato o tono Tareas con formato específico o dominio concreto Bajo
RAG Recuperar documentos relevantes y pasarlos al LLM como contexto Cuando necesitas que el modelo use información propia de la empresa Medio
Fine-tuning Reentrenar el LLM con datos propios para especializarlo Dominio muy específico, tono de marca, idioma técnico Alto
Agentes (Agents) El LLM usa herramientas externas (APIs, búsqueda, código) para completar tareas Automatización de flujos multi-paso Variable

RAG — la arquitectura más usada en empresas

Retrieval-Augmented Generation: en lugar de pedirle al LLM que "recuerde" información, le pasas los documentos relevantes en cada consulta. Así el modelo responde con tu información, no con lo que aprendió en su entrenamiento original.

Cómo funciona RAG
  • El usuario hace una pregunta
  • Se buscan los documentos más relevantes en una base vectorial
  • Esos documentos se incluyen en el prompt
  • El LLM responde basándose en ese contexto
  • Resultado: respuestas con tus datos, actualizables sin reentrenar
Componentes técnicos
  • Embeddings: representación numérica del texto para búsqueda semántica
  • Vector DB: base de datos especializada en búsqueda por similitud (Pinecone, Weaviate, pgvector)
  • Chunking: cómo se dividen los documentos en fragmentos
  • Context window: cuánto texto puede procesar el LLM a la vez

Agentes de AI

Un agente es un sistema donde el LLM no solo responde, sino que decide qué acciones tomar para completar una tarea: buscar en internet, ejecutar código, llamar a una API, escribir en una base de datos. El LLM actúa como el "cerebro" que orquesta herramientas.

Ejemplo práctico Un agente de soporte: el usuario reporta un problema → el agente busca en la documentación (RAG) → consulta el estado de la cuenta en el CRM (API tool) → genera una respuesta personalizada → si no puede resolverlo, escala a un humano. Todo sin intervención manual.

Evaluación de sistemas GenAI — diferente al ML clásico

Métrica

Faithfulness

¿La respuesta se basa en el contexto dado o el modelo "inventa" información? Crítico en RAG.

Métrica

Relevance

¿La respuesta es pertinente a la pregunta? El modelo puede ser fiel y aún así responder algo que no sirve.

Problema

Hallucination

El modelo genera información falsa pero con mucha confianza. El riesgo más crítico en aplicaciones de negocio.

Herramienta

LLM-as-Judge

Usar otro LLM para evaluar las respuestas del primero. Más escalable que evaluación humana, pero introduce sus propios sesgos.

Riesgo

Prompt Injection

Inputs maliciosos que hacen que el modelo ignore sus instrucciones. El equivalente a un SQL injection, pero para LLMs. Relevante en sistemas con agentes.

Práctica

Guardrails

Reglas y filtros que controlan qué puede y no puede generar el LLM. Críticos en apps de cliente final para evitar respuestas inapropiadas o riesgosas.

Para el manager Antes de hacer fine-tuning (caro), prueba siempre RAG primero. El 80% de los casos de uso empresarial se resuelve bien con RAG + buen prompting. Fine-tuning solo vale cuando tienes un dominio muy técnico, tono muy específico, o latencia crítica.

Roles en un equipo de AI

quién hace qué — y por qué hay tanta confusión con los títulos
Datos

Data Engineer

Construye y mantiene los pipelines de datos. Se asegura que los datos lleguen limpios, a tiempo, y en el formato correcto al data scientist. Sin él, no hay datos para entrenar.

Análisis

Data Analyst

Explora datos, construye dashboards, responde preguntas de negocio con datos históricos. No entrena modelos predictivos, pero entiende el negocio profundamente. Herramientas: SQL, Power BI, Tableau.

Modelos

Data Scientist

Diseña, entrena y evalúa modelos de ML. Combina estadística, programación, y entendimiento del negocio. En muchas empresas también hace parte del trabajo del DE y del ML Engineer.

Producción

ML Engineer

Lleva los modelos del data scientist a producción. Escribe código robusto, construye APIs, optimiza latencia. El puente entre ciencia y ingeniería.

Operaciones

MLOps Engineer

Se ocupa de la infraestructura de ML: pipelines automáticos, monitoreo, registros de modelos, CI/CD para ML. En equipos pequeños el ML Engineer asume este rol.

GenAI

AI/LLM Engineer

Especializado en sistemas con LLMs: diseña arquitecturas RAG, construye agentes, optimiza prompts, integra APIs de modelos. Rol nuevo y en alta demanda.

Diseño

Prompt Engineer

Optimiza cómo se le habla al LLM para obtener mejores resultados. Más relevante en producción que en prototipado. El rol está siendo absorbido por el LLM Engineer.

Estrategia

AI Product Manager

Define qué construir con AI, prioriza casos de uso, gestiona expectativas entre negocio y equipo técnico. Necesita entender tanto el problema como las limitaciones del modelo.

La confusión más común En startups, una persona hace todo: DE + DS + MLE. En empresas grandes están separados. Cuando contratas, sé específico: ¿necesitas a alguien que construya pipelines (DE), que entrene modelos (DS), o que los lleve a producción (MLE)? Son habilidades distintas.

Data Scientist vs ML Engineer — la distinción práctica

Data Scientist
  • Foco: explorar, experimentar, entender
  • Herramienta principal: Jupyter Notebook
  • Habla de accuracy, features, hipótesis
  • Output: un modelo que funciona en un notebook
  • Más formación en estadística y matemáticas
ML Engineer
  • Foco: construir, escalar, producción
  • Herramienta principal: código, cloud, APIs
  • Habla de latencia, throughput, reliability
  • Output: un modelo que corre en producción
  • Más formación en ingeniería de software

Glosario de términos AI/ML

lo que escuchas en reuniones y lo que realmente significa
FeatureVariable de entrada al modelo. Ej: "edad", "monto de la transacción", "hora del día". El modelo aprende a usar estas features para predecir.
Label / TargetLa variable que el modelo intenta predecir. En clasificación de fraude, el label es "fraude / no fraude".
HiperparámetroConfiguración del modelo que el humano define antes del entrenamiento (ej: número de capas, tasa de aprendizaje). Distinto de los parámetros que el modelo aprende solo.
EpochUna pasada completa del modelo por todos los datos de entrenamiento. Entrenar por muchas epochs puede llevar a overfitting.
Inference / InferenciaCuando el modelo ya entrenado recibe un nuevo dato y produce una predicción. Distinto del entrenamiento — en producción solo hacemos inference.
EmbeddingRepresentación numérica (vector) de texto, imágenes, u otros datos. Permite comparar similitud entre conceptos. La base de RAG y la búsqueda semántica.
TokenLa unidad mínima de texto que procesa un LLM. Aproximadamente ¾ de una palabra. Los modelos tienen un límite de tokens por consulta (context window).
Context WindowCuánto texto puede "ver" el LLM a la vez. Medido en tokens. GPT-4 Turbo: 128K tokens. Claude 3: hasta 200K. Determina cuántos documentos puedes meter en RAG.
TemperatureParámetro que controla qué tan "creativo" o "determinista" es el LLM. Temp = 0: siempre la misma respuesta. Temp = 1: más variedad. Para producción suele usarse baja.
HallucinationEl LLM genera información falsa pero presentada con confianza. No "sabe" que miente. El riesgo principal en aplicaciones empresariales.
Fine-tuningContinuar el entrenamiento de un modelo pre-entrenado con datos propios para especializarlo. Caro pero efectivo para dominios muy específicos.
RAGRetrieval-Augmented Generation. Dar al LLM documentos relevantes como contexto en cada consulta. Alternativa más barata al fine-tuning para incorporar conocimiento propio.
Vector DatabaseBase de datos optimizada para buscar documentos por similitud semántica. Compara embeddings. Ej: Pinecone, Weaviate, Chroma, pgvector. Componente central de RAG.
DriftData drift: los datos de entrada cambian con el tiempo. Concept drift: la relación entre inputs y outputs cambia (el mundo cambió). Ambos degradan el modelo.
BaselineEl punto de comparación mínimo. Si un modelo de fraude no supera a "siempre predecir no fraude", no sirve. El modelo debe superar el baseline para justificar su existencia.
Latency vs ThroughputLatencia: cuánto tarda en responder una sola consulta. Throughput: cuántas consultas puede procesar por segundo. Ambas importan en producción.
MLflow / W&BHerramientas de experiment tracking. Registran automáticamente qué se probó, con qué parámetros, y con qué resultados. El cuaderno de laboratorio del data scientist.
Batch vs Real-timeBatch: el modelo procesa muchos datos juntos (ej: cada noche). Real-time (online): el modelo responde al instante a cada consulta. Implican arquitecturas muy diferentes.
Ground truthLos datos correctos (etiquetados por humanos) contra los cuales se compara el modelo. Sin ground truth, no puedes medir si el modelo funciona.
Shadow modeCorrer el nuevo modelo en paralelo con el modelo actual, sin que sus predicciones afecten al usuario. Sirve para validar antes del cutover definitivo.

Quiz — Valida lo que absorbiste

7 preguntas sobre el ciclo completo y genai
Tu equipo dice "el modelo tiene 97% de accuracy detectando fraude". ¿Por qué deberías desconfiar de esa métrica sola?
En datos desbalanceados, accuracy engaña. El modelo necesita evaluarse con Recall (¿cuántos fraudes reales detectó?) y Precision (¿cuántas alertas eran fraude real?). Pide siempre la matriz de confusión.
¿Cuál es la diferencia entre fine-tuning y RAG para un proyecto con LLMs?
RAG es la primera opción para la mayoría de proyectos empresariales. Fine-tuning vale cuando el dominio es muy técnico, el volumen de consultas es masivo, o necesitas latencia mínima.
El data scientist entrega un modelo con excelentes métricas. Dos meses después, el desempeño cae sin que nadie cambie el código. ¿Qué ocurrió probablemente?
Los modelos se degradan aunque nadie toque el código. El mundo cambia, los usuarios cambian, los patrones cambian. Por eso el monitoreo y reentrenamiento periódico son parte del ciclo, no opcionales.
¿Qué es un "hallucination" en el contexto de LLMs?
Las alucinaciones son el riesgo principal en producción con LLMs. Mitigaciones: RAG (anclar respuestas en documentos reales), temperatura baja, guardrails, y evaluación continua de faithfulness.
Un proyecto de AI lleva 3 meses. El equipo dice "el modelo estará listo la próxima semana". Llevan dicho eso 4 semanas. ¿Cuál es la causa más probable?
El 60–80% del tiempo en AI se va en datos, no en el modelo. Si esto pasa, pregunta: ¿qué datos necesitamos que aún no tenemos? ¿El problema que queremos resolver está bien definido en términos de qué predice el modelo exactamente?
¿Cuál es la diferencia entre un Data Scientist y un ML Engineer?
En equipos pequeños una persona hace ambas. En equipos maduros están separados porque las habilidades son distintas: estadística vs ingeniería de software. Un gran modelo en un notebook no sirve sin el MLE que lo lleva a producción.
El equipo propone construir un "agente de AI" para automatizar el proceso de onboarding de clientes. ¿Qué pregunta es más crítica hacer primero?
Los agentes fallan. La pregunta crítica no es la tecnología sino la gobernanza: quién supervisa, cómo se detectan errores, y qué proceso existe para corregir. Un agente sin supervisión en un proceso crítico es un riesgo operacional.
